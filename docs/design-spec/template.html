    1.Overview & Goals

    This document specifies Quill, a small multi-author blogging platform: authors draft posts in a rich editor, editors review and schedule them, and readers browse a fast public site with comments and RSS. It is deliberately generic — the point is to exercise every part of the spec template, not to describe a real product.

    Goals

    • Author-friendly writing: autosaving drafts, a clean editor, and a preview that matches the live rendering exactly.
    • Editorial control: a lightweight review workflow (draft → in review → scheduled → published) with an audit trail of who changed what.
    • Fast public reads: statically-cacheable pages, no per-request database work on the hot path, and clean, shareable URLs.
    • Healthy comments: threaded discussion with spam defense and human moderation, never blocking a page render.

    Non-goals (first release)

    • Paid subscriptions, paywalls, or memberships.
    • A plugin/extension marketplace.
    • Multi-language content translation (the UI itself is direction-agnostic, but posts are single-language).
    Scope decision: the first release ships the author, editor, reader, and comment experiences end to end. Analytics beyond basic view counts, and a public write API, are explicit follow-ups (§10).

    2.Definitions

    Terms used throughout this document.

    TermMeaning
    PostA single article with a title, body (rich text), author, tags, and a lifecycle status. Identified publicly by a URL slug.
    RevisionAn immutable snapshot of a post's content, written on every save. The current post points at its latest revision; older ones power history and diffing.
    Author / EditorRoles. An author writes and submits their own posts; an editor can review, schedule, publish, and edit anyone's.
    CommentA reader-submitted reply attached to a post, optionally nested under a parent comment, with its own moderation state.
    SlugThe URL-safe identifier for a post (my-first-post), unique per site, derived from the title and editable before first publish.
    FeedA machine-readable list of recent published posts (RSS/Atom + JSON Feed), regenerated on publish.

    3.Domain Model & Schema

    Five core tables. Every table uses a UUID primary key. Content lives in revisions, not on posts, so history is free and the current body is just the newest revision.

    erDiagram
    USERS ||--o{ POSTS : authors
    USERS ||--o{ COMMENTS : writes
    POSTS ||--o{ REVISIONS : "has history"
    POSTS ||--o{ COMMENTS : receives
    POSTS }o--o{ TAGS : "labeled with"
    COMMENTS |o--o{ COMMENTS : "replies to"

    USERS {
        uuid id PK
        string email UK
        string name
        string role "author / editor"
    }
    POSTS {
        uuid id PK
        uuid author_id FK
        string title
        string slug UK
        string status "draft / in_review / scheduled / published / archived"
        uuid current_revision_id FK
        datetime published_at "nullable"
        int views_count
    }
    REVISIONS {
        uuid id PK
        uuid post_id FK
        uuid editor_id FK
        text body_html
        datetime created_at
    }
    COMMENTS {
        uuid id PK
        uuid post_id FK
        uuid parent_id FK "nullable, threading"
        string author_name
        text body
        string state "pending / approved / spam"
        datetime created_at
    }
    TAGS {
        uuid id PK
        string name UK
        string slug UK
    }

    Key decisions

    • Slug uniqueness is enforced by a unique index on posts.slug. Slugs are immutable after first publish so shared links never rot.
    • Status is a string-backed enum, stored as its own name (status = 'published') — self-describing and reorder-proof.
    • views_count is a counter cache incremented out-of-band (a queued job), never in the read path.
    • Comment threading is a self-referential parent_id; depth is capped in the UI, not the schema.

    4.Post Lifecycle

    A post moves through a small state machine. Authors drive the left side; editors drive publication. Every transition writes a revision and an audit entry.

    stateDiagram-v2
    [*] --> draft: author creates
    draft --> in_review: author submits
    in_review --> draft: editor requests changes
    in_review --> scheduled: editor sets a future time
    in_review --> published: editor publishes now
    scheduled --> published: schedule fires
    published --> archived: editor unpublishes
    archived --> draft: editor restores to edit
    published --> published: editor edits (new revision)
    RuleBehavior
    Who can publishEditors only. Authors submit to review; they cannot self-publish in the first release.
    SchedulingA scheduled post carries a future published_at; a cron job flips it to published when the time arrives and regenerates feeds.
    Editing published postsAllowed — it writes a new revision and points current_revision_id at it. The public URL is unchanged; caches are purged for that slug.
    ArchivingRemoves a post from listings and feeds but keeps the URL returning 410 Gone so links fail honestly rather than 404-ing silently.

    5.Publishing Flow

    Publishing is the one write path that must stay fast and side-effect-complete. It renders the post to cacheable HTML, updates the feeds, and warms the cache — all after the transaction commits, via jobs.

    flowchart TD
    A["Editor clicks Publish"] --> B{"Valid?
    title, slug, body present"}
    B -->|no| B2["Show inline errors"]
    B -->|yes| C["Transaction:
    status = published
    write revision + audit"]
    C --> D["Commit"]
    D --> E["Enqueue: render static HTML"]
    D --> F["Enqueue: regenerate feeds"]
    D --> G["Enqueue: purge CDN for /slug"]
    E --> H["Cache warmed"]
    F --> H
    G --> H
    Ordering matters: jobs run after commit, never inside the transaction. A job that reads the post before the commit lands would render stale content — so the publish action enqueues on the after_commit hook only.

    6.Comments & Moderation

    Comments post through a small pipeline that scores spam, stores the comment as pending, and lets an editor approve in one click. Rendering the post never waits on any of it.

    sequenceDiagram
    actor Reader
    participant Web as Web app
    participant Spam as Spam scorer
    participant DB as Database
    participant Mod as Editor

    Reader->>Web: Submit comment
    Web->>Spam: score(text, metadata)
    Spam-->>Web: score 0..1
    alt score >= 0.9
        Web->>DB: store state = spam
    else
        Web->>DB: store state = pending
        Web->>Mod: notify "1 comment to review"
    end
    Web-->>Reader: "Thanks — awaiting review"
    Mod->>DB: approve -> state = approved
    • Never inline-blocking: the spam score has a hard timeout; on timeout the comment is stored pending and a human decides.
    • Approved comments are baked into the post's cached HTML on approval, so readers get them without a dynamic query.
    • Threading depth is capped at 3 levels in the UI to keep mobile readable.

    7.Frontend & Routes

    Two surfaces: a public reading site (server-rendered, cacheable) and a private authoring app (interactive). They share nothing but the design tokens.

    Public routes

    RoutePage
    GET /Home — latest published posts, paginated.
    GET /posts/:slugA single post with its approved comments.
    GET /tags/:slugPosts for one tag.
    GET /feed.xml, /feed.jsonRSS/Atom and JSON Feed.

    Authoring routes

    GET /studioDashboard — your drafts and the review queue.
    GET /studio/posts/:id/editThe editor, with autosave and live preview.
    POST /studio/posts/:id/submitAuthor submits to review.
    POST /studio/posts/:id/publishEditor publishes or schedules.

    Conventions

    The reading site is plain server-rendered HTML with progressive enhancement; the studio is a single-page app. Both consume the same CSS custom properties, so a theme change is one variable swap — exactly like this document.

    8.Authentication & Authorization

    Only the studio needs auth; the public site is anonymous. Authentication is session-cookie based; authorization is two roles and a handful of rules.

    • Sessions: signed, http-only, same-site cookies. No third-party auth in the first release.
    • Roles: author and editor. Authors act on their own posts up to submission; editors act on all posts and own publication.
    • Rule of least surprise: the same policy object gates both the API and the rendered buttons — a control a user can't use is never shown.
    class PostPolicy
  def edit?    = owner? || editor?
  def submit?  = owner?
  def publish? = editor?
  def destroy? = editor?
end
    Irreversible action: deleting a post is editor-only and hard-deletes its revisions after a 30-day grace window. Archiving (reversible) is the default; deletion is behind a confirm dialog.

    9.SEO, Feeds & Performance

    • Canonical URLs & sitemaps: every post declares a canonical link; /sitemap.xml lists published posts and regenerates on publish.
    • Social cards: Open Graph + Twitter meta from the post's title, excerpt, and cover image.
    • Feeds are regenerated as a job on every publish/unpublish, and served from cache.
    • Performance budget: a post page ships under 100 KB of critical CSS/JS and renders from CDN cache; the database is touched only on cache miss.
    Because published pages are static HTML, the reading site survives a database outage in read-only form — a deliberate resilience property, not an accident.

    10.Testing Strategy

    Behavior-first tests, with the lifecycle state machine as the centerpiece.

    • Lifecycle: every allowed transition succeeds and writes a revision + audit entry; every forbidden one is rejected (author cannot publish, published cannot skip to draft without archiving).
    • Publishing side effects: jobs enqueue only after_commit; feeds and caches reflect the new post; archiving returns 410.
    • Comments: spam-score timeout falls back to pending; approval bakes the comment into cached HTML; threading depth cap holds.
    • Authorization: the policy matrix across both roles, for the API and the rendered controls.
    • Slugs & feeds: slug uniqueness and immutability after publish; feed output validates against the RSS/JSON Feed schemas.

    11.Rollout & Phases

    Sequential, independently shippable slices.

    1. Core domain: users, posts, revisions, the lifecycle state machine, and the audit trail (no UI).
    2. Authoring studio: editor with autosave + preview, the review queue, publish/schedule.
    3. Public site: home, post pages, tags, static rendering + CDN caching.
    4. Comments: submission pipeline, spam scoring, moderation, cached rendering.
    5. Feeds & SEO: RSS/Atom/JSON, sitemaps, social cards.
    6. Follow-ups: analytics dashboard, a public write API, scheduled digests.

    12.Decisions Log agreed

    QuestionDecision
    Where does post content live?In immutable revisions; the post points at the current one. History and diffing come free.
    Can authors publish their own posts?No — editors own publication in the first release. Revisit if teams ask for trusted authors.
    What happens to a deleted post's URL?Archive returns 410 Gone; hard delete only after a 30-day grace window.
    How are comments kept fast?Approved comments are baked into the post's cached HTML; the render path never queries them live.
    How do publish side effects run?As jobs on after_commit only, never inside the publish transaction.
